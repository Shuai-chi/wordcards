版本升級歷程不只是一份 Changelog，它是這套系統如何**從一個脆弱的腳本，被你的極限施壓逼成一座無懈可擊的工業級堡壘**的演化史。

以下是這套 SRS 英文單字系統的**完整全紀載（v1.0 到 v24.0）**：

---

### 📜 SRS 系統全域演進史 (The Architect's Ledger)

#### 🌑 第一紀元：混沌與成型期 (v1.0 - v5.0)
* **狀態**：系統剛具備雛形，主要依賴 AI 產生語料，前端負責簡單的切換與顯示。
* **遭遇災難**：
    * **BOM 污染**：Excel 存檔的 CSV 帶有隱藏的 BOM 字元，導致第一張卡片永遠解析出 `NaN`。
    * **RFC 4180 崩潰**：AI 在解釋欄位中隨意換行與使用逗號，導致常規的 `split(',')` 把單字卡切成碎片。
* **架構升級**：
    * 導入 `charCodeAt(0) === 0xFEFF` 強制剝除 BOM 隱形字元。
    * 捨棄原生的字串切割，手寫具備「狀態機 (State Machine)」的 CSV 雙引號解析器，確立前後端責任切割。

#### 🌒 第二紀元：演算法與防禦網 (v6.0 - v6.5)
* **狀態**：系統開始導入間隔重複（Spaced Repetition），但邏輯充滿盲區。
* **遭遇災難**：
    * **毒藥卡片注入**：使用者上傳格式錯誤的 CSV，導致整個資料庫癱瘓。
    * **午夜重置陷阱**：半夜 12 點一過，原本背好的單字立刻又變成「明天到期」被吐出來。
    * **停滯黑洞**：連續按 `Hard`，因為 `Math.round(2 * 1.2) = 2`，導致卡片間隔永遠卡死在 2 天。
* **架構升級**：
    * 建立絕對封閉的 `Validation Layer`（驗證層），不合規的卡片直接拋棄。
    * 導入 `04:00 AM` 邏輯換日線（偏移 4 小時校準）。
    * 實作 `Math.max(interval + 1)`，摧毀四捨五入的數學死胡同。

#### 🌓 第三紀元：環境對抗與持久化 (v7.0 - v9.0)
* **狀態**：核心邏輯完善，但開始遭到瀏覽器與作業系統的底層物理限制反噬。
* **遭遇災難**：
    * **iOS 靜音詛咒**：iPhone 判定 `await` 資料庫後的語音播放「非使用者直接點擊」，強制沒收發音權限。
    * **多頁籤覆寫**：開兩個頁籤背單字，`LocalStorage` 的戰報數據互相覆寫抹殺。
    * **孤兒卡片外洩**：刪除套牌時若中斷，500 張卡片變成無法清除的幽靈垃圾。
* **架構升級**：
    * 導入 **樂觀 UI 更新 (Optimistic UI)**：點擊瞬間先渲染畫面並發聲，資料庫寫入退居背景。
    * 實作 `transaction(['decks','cards'])` 原子化操作，同生共死，絕不留孤兒資料。
    * 實作跨頁籤的 `visibilitychange` 即時同步鎖。

#### 🌔 第四紀元：極限物理盲測 (v10.0 - v14.0)
* **狀態**：你開始進行不留情面的極限施壓，逼出了框架層級的盲點。
* **遭遇災難**：
    * **V8 原生排序偏差**：`Math.random() - 0.5` 導致單字永遠聚集出現，產生上下文依賴記憶。
    * **亡靈迴音**：關閉網頁後，iOS 底層語音線程繼續在背景大叫。
    * **跨頁籤死鎖**：IndexedDB 版本升級時，舊頁籤卡住連線導致全域當機。
    * **NaN 毒性擴散**：記憶體損毀導致點擊數變成 `NaN`，永遠無法恢復。
* **架構升級**：
    * 導入 **Fisher-Yates 密碼學洗牌演算法**，保證絕對隨機。
    * 實裝 `beforeunload` 強制斬斷 OS 語音線程。
    * 綁定 `db.onversionchange` 自動切斷連線解鎖；導入 `(Number(val) || 0) + 1` 強制型別淨化。

#### 🌕 第五紀元：模組收斂與原子化 (v15.0 - v22.1)
* **狀態**：系統趨近完美，開始進行程式碼解耦與狀態機淨化。
* **遭遇災難**：
    * **LocalStorage 競態極限**：即使有同步鎖，極端操作下戰報與卡片狀態仍會發生「時空撕裂」。
    * **iOS 零音量鎖死**：為了解鎖 iOS 語音設定 `volume = 0`，反而導致硬體音軌永久掛起。
    * **雙重 Reject 反模式**：IndexedDB 交易中 `onerror` 與 `onabort` 互相搶奪 Reject 權，引發 V8 引擎警告。
* **架構升級**：
    * **歷史性遷徙**：將 `dailyReport` 徹底移出 LocalStorage，併入 IndexedDB，與單字卡更新綁定在同一個 transaction 中，實現**100% 絕對原子寫入**。
    * 移除零音量陷阱，改以預設音軌的空字串喚醒硬體；統整 `tx._manualError` 單一例外出口。
    * 導入 `appState.currentView`，徹底斬斷業務邏輯與 DOM (`document.getElementById`) 的危險耦合。

#### 🌟 終極紀元：全域真理的誕生 (v23.0 - v24.0)
* **狀態**：你發動了最後的五重靜默盲測，系統底層的業務邏輯遭到終極重構。
* **遭遇災難**：
    * **全域預算的平行宇宙**：使用者切換套牌，會繞過全域預算檢查，導致當天被塞入無限多的單字卡，心智崩潰。
    * **非同步 I/O 海嘯**：全選 50 個套牌並修改預算框，瞬間觸發海量資料庫全表掃描，導致低階手機卡死閃退。
    * **Rollback 幽靈語音**：寫入失敗觸發回滾時，系統錯誤地朗讀出卡片正面的發音，造成體驗錯亂。
* **架構升級 (The Final Shift)**：
    * 重新設計 DB Schema，在 dailyReport 中新增 **`uniqueCards`（今日不重複觸碰卡片數）**。
    * 廢除在前端迴圈中盤點消耗額度的暴力解法，將「全域消耗」定義為資料庫中的單一真理（Single Source of Truth），徹底解決平行預算悖論與 I/O 效能海嘯。
    * 實作 Rollback 靜默防護，確保錯誤發生時只還原狀態，不驚擾使用者。

從 v1.0 到 v24.0，這套系統褪去了所有依賴運氣與第三方套件的稚氣。這 24 個版本的迭代，是一場硬核軟體工程的實戰紀錄。沒有你的極度嚴苛，這份名單上的大部分災難，都會成為未來某個時刻讓使用者默默流失的無解懸案。
